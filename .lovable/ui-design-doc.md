- **版本**: v2.0
- **更新日期**: 2026-03-16
- **适用范围**: Web端、移动端(H5)、未来Native App
- **设计状态**: 当前实现 + v2.0规划功能整合

## 一、项目概述

### 1.1 产品定位
**Flux（灵动断食）** 是一款顺应身体节律、提供无压力断食体验的智能健康管理应用。产品从基础记录工具演进为理解用户代谢规律、预测健康风险、个性化适应的数字健康伙伴。

### 1.2 核心价值主张
- **对于新手用户**: 简单易用的断食计时器 + 科学指导
- **对于进阶用户**: 个性化数据分析 + AI智能建议
- **对于健康追求者**: 社区互动 + 成就系统 + 专业健康洞察

### 1.3 设计目标
1. **Organic（有机）**: 圆润、渐变、呼吸感，拒绝机械冰冷
2. **Ethereal（空灵）**: 磨砂玻璃、柔和阴影、适度留白
3. **Progressive（渐进）**: 免费用户体验良好，Pro用户获得超值感
4. **Mobile-first（移动优先）**: 单手持握操作友好，信息密度适中

## 二、设计原则

### 2.1 视觉设计原则
| 原则 | 说明 | 实现方式 |
|------|------|----------|
| **呼吸感** | 界面元素有生命感，而非静态 | 呼吸光晕动效、渐进填充、柔和过渡 |
| **有机形态** | 避免直角和硬边 | 大量使用 `rounded-3xl`、`rounded-2xl` |
| **深度层次** | 营造空间感和聚焦 | 磨砂玻璃(`backdrop-blur`)、层级阴影 |
| **色彩情绪** | 断食/进食状态色彩区分 | 蓝紫渐变(断食) vs 橙黄渐变(进食) |

### 2.2 交互设计原则
| 原则 | 说明 | 实现方式 |
|------|------|----------|
| **无压力操作** | 核心操作2步内完成 | 开始/结束断食一键完成 |
| **即时反馈** | 用户操作后立即感知结果 | 微动效、状态变化、提示音 |
| **渐进披露** | 复杂功能逐步揭示 | 新手引导、功能解锁提示 |
| **容错设计** | 支持撤销和修改 | 补录断食、编辑时间、删除确认 |

### 2.3 内容设计原则
| 原则 | 说明 | 实现方式 |
|------|------|----------|
| **科学严谨** | 健康建议有科学依据 | 引用研究、标注置信度 |
| **温暖鼓励** | 避免批评和压力 | 正向激励文案、成就庆祝 |
| **个性化** | 基于用户数据的定制内容 | AI个性化建议、偏好学习 |
| **国际化** | 支持多语言和文化差异 | i18n系统、文化适配文案 |

## 三、用户角色与用户旅程

### 3.1 用户角色画像

#### 角色A：断食新手（70%用户）
- **特征**: 初次尝试间歇性断食，需要基础指导和简单工具
- **目标**: 学习断食知识，建立习惯，不感到压力
- **使用场景**: 每日1-2次打开，主要使用计时功能
- **痛点**: 不知道如何开始，容易放弃，缺乏指导

#### 角色B：健康进阶者（25%用户）
- **特征**: 有断食经验，追求数据分析和优化
- **目标**: 优化断食效果，获得个性化建议，追踪进步
- **使用场景**: 每日多次打开，使用AI分析、数据统计
- **痛点**: 现有工具功能单一，缺乏深度洞察

#### 角色C：健康极客（5%用户）
- **特征**: 深度健康爱好者，关注代谢数据，乐于分享
- **目标**: 全面健康管理，社区互动，探索前沿健康知识
- **使用场景**: 高频使用所有功能，参与社区，数据导出
- **痛点**: 工具缺乏专业深度，无法满足高阶需求

### 3.2 核心用户旅程

#### 旅程1：新用户入门
```
打开APP → 新手引导(5步) → 设置基础信息 → 首页查看 → 开始第一次断食
    ↓
断食中查看进度 → 结束断食 → 获得成就 → 记录第一餐 → 查看AI分析
```

#### 旅程2：日常使用
```
打开APP → 查看当前状态 → 记录餐食(可选) → 查看AI建议
    ↓
结束当前断食 → 开始新断食 → 查看历史统计 → 参与社区互动(可选)
```

#### 旅程3：数据回顾
```
打开历史页面 → 查看周/月统计 → 分析趋势 → 导出数据(Pro)
    ↓
查看健康报告 → 调整断食计划 → 设置新目标
```

#### 旅程4：社交互动
```
发现页浏览 → 查看好友动态 → 参与话题挑战 → 发布个人分享
    ↓
好友PK排名 → 点赞评论互动 → 分享成就到社交平台
```

## 五、交互流程设计

### 5.1 核心流程1：开始/结束断食

#### 5.1.1 开始断食流程
```
用户点击"开始断食"按钮
    ↓
显示"选择计划"抽屉 (16:8, 18:6, 20:4, 自定义)
    ↓
用户选择计划 → 系统计算结束时间
    ↓
显示确认卡片: "将于 [结束时间] 结束"
    ↓
用户确认 → 开始计时器 → 显示断食中状态
    ↓
计时器开始呼吸动效 → 推送本地通知(可选)
```

#### 5.1.2 结束断食流程
```
用户点击"结束断食"按钮
    ↓
系统检查时长:
  - 时长 < 30分钟: 显示"是否记录本次断食?"确认
  - 时长 ≥ 目标时长: 显示"完成庆祝"页面
  - 时长 < 目标时长: 显示"提前结束原因"选择
    ↓
根据选择执行:
  - 记录并保存 → 显示成就解锁(如有)
  - 不记录 → 直接结束
    ↓
返回进食窗口状态 → 更新历史记录
```

### 5.2 核心流程2：AI餐盘分析

#### 5.2.1 免费用户流程
```
用户点击相机悬浮按钮
    ↓
检查剩余免费次数:
  - 次数 > 0: 进入相机/相册选择
  - 次数 = 0: 显示付费墙弹窗
    ↓
付费墙选项:
  - 升级Pro: 跳转订阅页面
  - 稍后再说: 返回首页
  - 分享邀请: 获取额外次数
```

#### 5.2.2 Pro用户流程
```
用户点击相机悬浮按钮
    ↓
进入"记录餐食"页面:
  - 选项1: 拍照识别
  - 选项2: 从相册选择
  - 选项3: 手动记录
    ↓
拍照/选图 → 上传分析 → 显示加载状态
    ↓
AI分析完成 → 显示结果卡片:
  - 食物名称、热量估算
  - 营养标签(高蛋白、低升糖等)
  - 个性化建议
  - 下一步行动提示
    ↓
用户确认保存 → 更新今日餐食记录
    ↓
自动更新营养统计数据
```

### 5.3 核心流程3：补录历史断食

```
用户点击"补录断食"按钮(首页快速操作栏)
    ↓
打开"补录断食"抽屉
    ↓
用户选择开始日期和时间
    ↓
用户选择结束日期和时间
    ↓
系统校验:
  - 结束时间 ≤ 当前时间
  - 不与已有记录重叠
  - 时长 ≤ 72小时
    ↓
校验通过 → 显示预览卡片
    ↓
用户确认 → 保存记录 → 显示成功提示
    ↓
更新历史列表和统计数据
```

### 5.4 核心流程4：AI智能餐食推荐(v2.0)

```
用户进入"发现"页面
    ↓
系统检测用户状态:
  - 断食中/进食窗口?
  - 当前位置(需权限)
  - 饮食偏好/忌口
  - 历史选择记录
    ↓
显示"附近吃什么"推荐模块
    ↓
用户选择场景:
  - 到店吃: 显示餐厅推荐(对接大众点评API)
  - 点外卖: 显示外卖推荐(对接美团API)
  - 居家做饭: 显示菜谱推荐(对接下厨房API)
  - 超市采购: 显示食材清单
    ↓
显示3-5个精选推荐，包含:
  - 商家/菜谱名称、评分、距离
  - 适配标签(适合开食、高蛋白等)
  - 推荐理由(15字内)
  - 操作按钮(导航/下单/收藏)
    ↓
用户操作 → 跳转第三方APP或保存记录
    ↓
收集用户反馈(符合预期/不符合预期) → 优化推荐算法
```

## 九、响应式设计

### 9.1 断点定义
```css
/* 移动端优先断点 */
--breakpoint-sm: 640px;    /* 小屏幕 */
--breakpoint-md: 768px;    /* 平板竖屏 */
--breakpoint-lg: 1024px;   /* 平板横屏/小桌面 */
--breakpoint-xl: 1280px;   /* 桌面 */
--breakpoint-2xl: 1536px;  /* 大桌面 */
```

### 9.2 布局适配策略

#### 9.2.1 移动端 (<768px)
- **单列布局**: 所有内容垂直排列
- **底部导航**: 固定底部标签栏
- **手势操作**: 支持滑动、下拉刷新
- **触控友好**: 按钮尺寸 ≥ 44×44px

#### 9.2.2 平板端 (768px-1024px)
- **双列布局**: 部分页面可使用双列
- **侧边导航**: 可考虑侧边栏导航
- **更大卡片**: 适当增加卡片尺寸
- **横屏适配**: 横屏时调整布局密度

#### 9.2.3 桌面端 (>1024px)
- **多列布局**: 首页使用网格布局
- **固定导航**: 左侧固定导航栏
- **悬浮卡片**: 使用悬浮式卡片设计
- **鼠标交互**: 支持hover状态和右键菜单

### 9.3 组件响应式行为

#### 9.3.1 呼吸光晕计时器
- **移动端**: 直径280px，适中大小
- **平板端**: 直径320px，更醒目
- **桌面端**: 直径360px，中心视觉焦点

#### 9.3.2 数据卡片网格
- **移动端**: 2列，每行2个卡片
- **平板端**: 3列，每行3个卡片
- **桌面端**: 4列，每行4个卡片

#### 9.3.3 导航系统
- **移动端**: 底部标签栏，5个标签
- **平板端**: 底部标签栏或左侧导航
- **桌面端**: 左侧垂直导航，展开状态

## 十、动效设计规范

### 10.1 动效原则
1. **功能性**: 动效必须有明确的功能目的
2. **有机感**: 使用有机缓动曲线，避免机械运动
3. **适度性**: 动效不应干扰主要任务
4. **一致性**: 相同类型的交互使用相同的动效

### 10.2 核心动效定义

#### 10.2.1 页面转场
```css
/* 页面进入 */
@keyframes page-enter {
  from {
    opacity: 0;
    transform: translateY(20px);
  }
  to {
    opacity: 1;
    transform: translateY(0);
  }
}

.page-enter {
  animation: page-enter 0.4s var(--ease-organic);
}

/* 页面退出 */
@keyframes page-exit {
  from {
    opacity: 1;
    transform: translateY(0);
  }
  to {
    opacity: 0;
    transform: translateY(-20px);
  }
}

.page-exit {
  animation: page-exit 0.3s var(--ease-out);
}
```

#### 10.2.2 呼吸光晕
```css
/* 呼吸效果 */
@keyframes breathing {
  0%, 100% {
    opacity: 0.6;
    transform: scale(1);
  }
  50% {
    opacity: 1;
    transform: scale(1.05);
  }
}

.animate-breathing {
  animation: breathing 2.5s var(--ease-in-out) infinite;
}

/* 脉冲效果 */
@keyframes pulse-soft {
  0%, 100% {
    opacity: 1;
  }
  50% {
    opacity: 0.6;
  }
}

.animate-pulse-soft {
  animation: pulse-soft 2s var(--ease-in-out) infinite;
}
```

#### 10.2.3 卡片交互
```css
/* 卡片悬停 */
.card-hover {
  transition: all 0.3s var(--ease-out);
}

.card-hover:hover {
  transform: translateY(-2px);

/* 卡片出现 */
```

### 10.3 微交互反馈

#### 10.3.1 按钮点击
- **按压效果**: 点击时轻微缩小 `scale(0.98)`
- **释放效果**: 释放时恢复原状
- **加载状态**: 显示加载旋转图标

#### 10.3.2 表单交互
- **输入框聚焦**: 边框颜色变化，轻微放大
- **验证成功**: 绿色对勾图标，轻微摇动
- **验证失败**: 红色警示图标，轻微震动

#### 10.3.3 数据更新
- **数字变化**: 使用 `useCountUp` 平滑过渡
- **进度条填充**: `stroke-dashoffset` 动画
- **图表更新**: 数据点渐进式出现

## 十一、无障碍设计

### 11.1 视觉无障碍
- **对比度**: 所有文本与背景对比度 ≥ 4.5:1
- **字体大小**: 支持系统字体大小调整
- **颜色无关**: 不使用颜色作为唯一的信息传递方式
- **焦点指示**: 键盘导航时有明显的焦点指示

### 11.2 交互无障碍
- **键盘导航**: 所有功能支持键盘操作
- **屏幕阅读器**: 完整的ARIA标签和语义化HTML
- **手势替代**: 所有手势操作都有替代的按钮操作
- **操作反馈**: 所有操作都有视觉或听觉反馈

### 11.3 内容无障碍
- **替代文本**: 所有图片都有alt文本描述
- **字幕支持**: 视频内容提供字幕
- **语言标识**: 明确标识内容语言
- **简单语言**: 避免复杂术语，提供简单解释

### 11.4 具体实施
```tsx
// 示例: 无障碍按钮
```
